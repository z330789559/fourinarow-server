这是一个“方案规划/可行性分析”请求，不需要改代码。基于当前仓库结构（Actix actor + MongoDB + 单进程内存状态），可按下面清单改造：

改造总目标（微信/抖音小游戏后端化）

从“对战房间服务”升级为“玩家中心 + 世界大厅 + 可选房间 + 运营系统”。
保留现有 websocket 实时链路与房间能力，新增玩家资产、任务、排行榜、定时事件、批处理持久化。
难度：中高；可执行性：高（分阶段做）。
一、玩家与资产模型（Player Domain）

新增玩家主档：基础信息、成长属性、背包/货币、体力、统计计数器。
新增资产子域：可堆叠道具、软硬货币、消耗与发放流水、幂等事务号。
将现有 user/game_info 扩展为“可持续迭代”的玩家聚合，而不只是对战 SR。
难度：中；可执行性：高（现有 users 集合可平滑扩展）。
二、player.get()/set() 抽象层

在业务层引入统一 PlayerRepository 接口：读策略、写策略、缓存命中策略统一收口。
所有上层逻辑（任务、排行榜、奖励）只通过 player.get/set 操作玩家聚合，避免直接散落 DB 调用。
难度：中；可执行性：高（当前 UserManager/DatabaseManager 结构可承载）。
三、dirty flag + write-behind + 批量落库

在在线玩家内存态引入 dirty 标记（按字段分桶：资产、任务、统计、社交）。
引入异步写回队列，按时间窗/条数阈值批量 flush 到 Mongo。
增加“下线强制刷盘 + 周期刷盘 + 进程关闭刷盘 + 失败重试/死信补偿”。
增加版本号/乐观锁，避免并发覆盖。
难度：高；可执行性：中高（当前是“即时 update_one”，改为写后延迟需要重构数据一致性路径）。
四、任务系统（主线/成就/每日）

主线任务：一次性状态机（未接取/进行中/完成/已领奖）。
成就任务：模板驱动 + 阶梯目标（10/100/1000）可叠加实例化。
每日任务：按日重置，支持随机池/权重池。
任务进度事件化：击杀、登录、在线时长、货币消耗等统一事件源更新。
奖励中心统一发奖（防重复、幂等）。
难度：中高；可执行性：高（当前尚无任务模块，但架构可新增独立 domain）。
五、延时通知与定时活动封装（如 12 点活动）

抽象“调度中心”：一次性触发、周期触发、日历触发（按时区）。
活动通知支持：全服广播、分群推送、玩家个人 inbox。
需要可恢复调度（服务重启后继续），并记录执行日志与补偿策略。
难度：中高；可执行性：中高（当前有 run_interval 但仅进程内临时定时，不是可恢复任务调度）。
六、排行榜 + 抽奖模块

排行榜：多榜单（战力/关卡/击杀/活动分），支持周期结算与快照归档。
抽奖：奖池配置、概率版本、保底规则、风控限制、审计日志。
与任务/活动联动：活动积分榜结算后触发发奖。
难度：中高；可执行性：高（可先 Mongo 版本，后续接 Redis 提升性能）。
七、世界大 Lobby + Room 共存（你说的 flybirds 场景）

保留 room（双人合作/PK）能力。
新增 world lobby：非对战玩家常驻、社交/活动/匹配入口都在大 lobby。
路由策略：单人玩法走 world session，组队/PK 按需进入 room，再返回 world。
难度：中；可执行性：高（现有 LobbyManager 可扩展为 WorldManager + RoomManager 双层）。
八、平台化能力（微信/抖音必须补）

接入平台登录鉴权（openid/unionid/session）与账号绑定。
支付/订单校验、风控、封禁、申诉审计。
灰度配置、公告配置、运营开关、A/B 配置。
难度：高；可执行性：中高（与业务域并行推进）。
九、分阶段落地建议（按风险最小化）

Phase 1：玩家聚合 + player.get/set + 资产流水（先不 write-behind）。
Phase 2：任务系统（主线/成就/每日）+ 奖励中心。
Phase 3：排行榜 + 活动调度 + 延时通知。
Phase 4：dirty/write-behind/批量落库 + 可恢复补偿。
Phase 5：world lobby + room 共存重构 + 平台登录/支付/风控。
难度曲线：前低后高；可执行性：高（可逐步上线）。
十、对“当前项目”的总体结论

可作为“实时连接与房间框架”基础继续演进。
真正的小游戏后端核心能力（玩家资产、任务、调度、排行榜、运营化）目前基本空白，需要新增 domain。
最大技术挑战是从“直接写库”过渡到“内存态 + dirty + write-behind + 一致性保障”。
